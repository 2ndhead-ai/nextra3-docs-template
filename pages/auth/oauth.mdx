# OAuth Implementation in Express.js

OAuth 2.0 is a powerful authentication and authorization framework that allows third-party applications to access user resources without exposing credentials. This guide covers everything you need to implement OAuth 2.0 in your Express.js applications.

## OAuth 2.0 Concepts and Flow

### What is OAuth 2.0?

OAuth 2.0 is an industry-standard protocol for authorization that enables applications to secure designated access to user accounts on an HTTP service. Instead of sharing passwords, OAuth uses access tokens to authorize applications.

### Key Roles in OAuth

- **Resource Owner**: The user who owns the data (typically your application's user)
- **Client**: The application requesting access to the user's data
- **Authorization Server**: The server that authenticates the user and issues tokens
- **Resource Server**: The server hosting the protected user data

### OAuth 2.0 Flow Types

1. **Authorization Code Grant**
   - Most commonly used for server-side web applications
   - Provides the most secure flow with refresh tokens
   - Involves redirecting the user to the provider, then back to your app with a code
   - Code is exchanged for access and refresh tokens

2. **Implicit Grant**
   - Used for browser-based applications without a server component
   - Returns access token directly after authorization
   - No refresh tokens, shorter token lifetimes
   - Less secure than authorization code flow

3. **Client Credentials Grant**
   - Used for server-to-server communication
   - The application acts on its own behalf, not on behalf of a user

4. **Resource Owner Password Credentials Grant**
   - Application collects username and password directly
   - Should only be used when other flows aren't viable
   - Higher risk as it exposes user credentials to the application

### OAuth 2.0 Flow Overview

The typical OAuth 2.0 authorization code flow works as follows:

1. User initiates login with a provider (e.g., "Login with Google")
2. Your application redirects to the provider's authorization server
3. User authenticates with the provider and grants permissions
4. Provider redirects back to your application with an authorization code
5. Your application exchanges the code for access and refresh tokens
6. Your application uses the access token to request user information
7. The information is used to authenticate/register the user in your system

## Integrating OAuth with Popular Providers

### Setting Up OAuth with Google

First, set up your credentials in the Google Developer Console:

1. Create a project in the Google Developer Console
2. Configure the OAuth consent screen
3. Create OAuth 2.0 client credentials
4. Note your Client ID and Client Secret
5. Configure authorized redirect URIs (e.g., `http://localhost:3000/auth/google/callback`)

### Setting Up OAuth with GitHub

1. Register a new OAuth application in GitHub Developer Settings
2. Provide your application name and homepage URL
3. Set the authorization callback URL
4. Receive your Client ID and Client Secret

### Setting Up OAuth with Facebook

1. Create a new app in the Facebook Developer Portal
2. Add the Facebook Login product
3. Configure valid OAuth redirect URIs
4. Note your App ID and App Secret

## Implementation with Passport.js

Passport.js is an authentication middleware for Node.js that integrates perfectly with Express.js and supports numerous authentication strategies, including OAuth providers.

### Step 1: Install Required Packages

const express = require('express');
const session = require('express-session');
const passport = require('passport');
const GoogleStrategy = require('passport-google-oauth20').Strategy;
const GitHubStrategy = require('passport-github2').Strategy;
require('dotenv').config();

### Step 2: Configure Environment Variables

Create a `.env` file to store OAuth credentials:

SESSION_SECRET=your_secure_session_secret
GOOGLE_CLIENT_ID=your_google_client_id
GOOGLE_CLIENT_SECRET=your_google_client_secret
GITHUB_CLIENT_ID=your_github_client_id
GITHUB_CLIENT_SECRET=your_github_client_secret

### Step 3: Configure Express and Passport

const app = express();

// Session configuration
app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: { secure: process.env.NODE_ENV === 'production' }
}));

// Initialize Passport
app.use(passport.initialize());
app.use(passport.session());

// Serialize and deserialize user (for session support)
passport.serializeUser((user, done) => {
  done(null, user);
});

passport.deserializeUser((obj, done) => {
  done(null, obj);
});

### Step 4: Configure OAuth Strategies

#### Google OAuth Strategy

passport.use(new GoogleStrategy({
    clientID: process.env.GOOGLE_CLIENT_ID,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET,
    callbackURL: "/auth/google/callback"
  },
  function(accessToken, refreshToken, profile, done) {
    // In a real application, you would find or create a user in your database
    // For this example, we'll just use the profile info
    const user = {
      id: profile.id,
      displayName: profile.displayName,
      email: profile.emails[0].value,
      provider: 'google',
      photo: profile.photos[0].value,
      accessToken
    };
    
    return done(null, user);
  }
));

#### GitHub OAuth Strategy

passport.use(new GitHubStrategy({
    clientID: process.env.GITHUB_CLIENT_ID,
    clientSecret: process.env.GITHUB_CLIENT_SECRET,
    callbackURL: "/auth/github/callback"
  },
  function(accessToken, refreshToken, profile, done) {
    // In a real application, you would find or create a user in your database
    const user = {
      id: profile.id,
      displayName: profile.displayName || profile.username,
      username: profile.username,
      provider: 'github',
      photo: profile.photos?.[0]?.value,
      accessToken
    };
    
    return done(null, user);
  }
));

### Step 5: Create Authentication Routes

#### Google Authentication Routes

// Route to initiate Google OAuth
app.get('/auth/google', 
  passport.authenticate('google', { scope: ['profile', 'email'] })
);

// Callback route after Google OAuth
app.get('/auth/google/callback', 
  passport.authenticate('google', { failureRedirect: '/login' }),
  (req, res) => {
    // Successful authentication
    res.redirect('/profile');
  }
);

#### GitHub Authentication Routes

// Route to initiate GitHub OAuth
app.get('/auth/github',
  passport.authenticate('github', { scope: ['user:email'] })
);

// Callback route after GitHub OAuth
app.get('/auth/github/callback',
  passport.authenticate('github', { failureRedirect: '/login' }),
  (req, res) => {
    // Successful authentication
    res.redirect('/profile');
  }
);

#### Logout Route

app.get('/logout', (req, res) => {
  req.logout();
  res.redirect('/');
});

### Step 6: Create Protected Routes

// Middleware to check if user is authenticated
function isAuthenticated(req, res, next) {
  if (req.isAuthenticated()) {
    return next();
  }
  res.redirect('/login');
}

// Protected profile route
app.get('/profile', isAuthenticated, (req, res) => {
  res.json({
    user: req.user,
    authenticated: true
  });
});

// Public route
app.get('/', (req, res) => {
  res.send(`
    <h1>OAuth Authentication Example</h1>
    <p>Login with:</p>
    <ul>
      <li><a href="/auth/google">Google</a></li>
      <li><a href="/auth/github">GitHub</a></li>
    </ul>
  `);
});

### Handling OAuth State Parameter

To prevent CSRF attacks, always use the state parameter:

app.get('/auth/google', (req, res, next) => {
  const state = crypto.randomBytes(16).toString('hex');
  req.session.authState = state;
  
  passport.authenticate('google', { 
    scope: ['profile', 'email'],
    state: state
  })(req, res, next);
});

app.get('/auth/google/callback', (req, res, next) => {
  const receivedState = req.query.state;
  const originalState = req.session.authState;
  
  if (!receivedState || !originalState || receivedState !== originalState) {
    return res.status(403).send('State validation failed');
  }
  
  // Delete the state from session to prevent reuse
  delete req.session.authState;
  
  passport.authenticate('google', { 
    failureRedirect: '/login' 
  })(req, res, next);
}, (req, res) => {
  res.redirect('/profile');
});

### Handling OAuth Errors

Add error handling for OAuth failures:

app.get('/auth/google/callback',
  (req, res, next) => {
    passport.authenticate('google', (err, user, info) => {
      if (err) {
        return res.redirect('/login?error=server_error');
      }
      
      if (!user) {
        return res.redirect(`/login?error=${info.message || 'authentication_failed'}`);
      }
      
      req.logIn(user, (err) => {
        if (err) {
          return res.redirect('/login?error=login_failed');
        }
        return res.redirect('/profile');
      });
    })(req, res, next);
  }
);

## Security Considerations and Best Practices

### 1. Never Store Access Tokens in Client-Side Storage

Access tokens should never be stored in localStorage or sessionStorage in browser applications.

Use secure, httpOnly cookies or server-side sessions.

### 2. Validate OAuth State Parameter

Always use and validate the state parameter to prevent CSRF attacks.

### 3. Implement PKCE for Mobile and SPA Applications

PKCE (Proof Key for Code Exchange) enhances security for public clients:

// Generate code verifier and challenge
const codeVerifier = crypto.randomBytes(32).toString('base64')
  .replace(/\+/g, '-')
  .replace(/\//g, '_')
  .replace(/=/g, '');

const codeChallenge = crypto
  .createHash('sha256')
  .update(codeVerifier)
  .digest('base64')
  .replace(/\+/g, '-')
  .replace(/\//g, '_')
  .replace(/=/g, '');

// Store code verifier in session
req.session.codeVerifier = codeVerifier;

// Use code challenge in authorization request
app.get('/auth/google',
  passport.authenticate('google', {
    scope: ['profile', 'email'],
    state: state,
    codeChallenge: codeChallenge,
    codeChallengeMethod: 'S256'
  })
);

// Include verifier when exchanging code for tokens
// (Actual implementation varies by provider and library)

### 4. Secure Sensitive Configuration

Never commit OAuth secrets to version control:

- Store credentials in environment variables
- Use .env files for development (add to .gitignore)
- Use secure secret management in production (e.g., AWS Secrets Manager, HashiCorp Vault)

### 5. Consider Token Storage and Refresh

For production applications, consider:

- Using a database to store refresh tokens
- Implementing secure token refresh procedures
- Revoking tokens when users log out
- Rotating tokens periodically

### 6. Enforce Proper Scope Management

Only request the OAuth scopes your application actually needs.

### 7. Implement Rate Limiting

Protect OAuth endpoints from brute force and DoS attacks:

const rateLimit = require('express-rate-limit');

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 10, // limit each IP to 10 requests per windowMs
  message: 'Too many authentication attempts, please try again later'
});

app.use('/auth', authLimiter);

### 8. Verify Email Addresses

For higher security applications, verify email addresses even when using OAuth:

// After successful OAuth authentication
if (!user.emailVerified) {
  // Send verification email
  sendVerificationEmail(user.email);
  return res.redirect('/verify-email-notice');
}

## Advanced OAuth Patterns

### OAuth for API Access

To use OAuth for API access:

1. **Define API access scopes**:
   - Create custom scopes for different API resources
   - Request these scopes during OAuth flow

2. **Token introspection**:
   - Verify tokens with the authorization server
   - Check token validity and granted scopes

3. **Protected API endpoints**:
   - Validate OAuth tokens for API requests
   - Check sufficient scopes for the requested operation

### Multi-Provider Authentication

Support multiple OAuth providers with a unified user model:

// After OAuth callback
async function handleOAuthCallback(req, res, next, provider, profile) {
  try {
    // Find existing user account connected to this OAuth provider
    let user = await User.findOne({
      [`${provider}Id`]: profile.id
    });
    
    if (user) {
      // User exists, log them in
      return finishAuth(req, res, user);
    }
    
    // If user is logged in, connect this OAuth account to their profile
    if (req.user) {
      req.user[`${provider}Id`] = profile.id;
      await req.user.save();
      return res.redirect('/profile?message=account_linked');
    }
    
    // Create a new user
    user = new User({
      [`${provider}Id`]: profile.id,
      email: profile.email,
      name: profile.displayName,
      // ... other user properties
    });
    await user.save();
    
    finishAuth(req, res, user);
  } catch (error) {
    next(error);
  }
}

function finishAuth(req, res, user) {
  req.login(user, (err) => {
    if (err) return next(err);
    res.redirect('/profile');
  });
}

## Troubleshooting Common OAuth Issues

### Invalid Redirect URI

Ensure your callback URL exactly matches what's registered with the provider.

### Missing or Invalid Scopes

Check that you've requested all necessary scopes for the data you need.

### Token Expiration Issues

Implement proper refresh token handling to maintain access.

### State Mismatch Errors

Verify state parameter handling and session persistence.

### Cross-Origin Issues

Ensure your OAuth redirects work properly with your application's domain structure.
